# Portal — login & personal file upload (`/portal/`)

Static page served at `https://www.theprivilegedcompany.com/portal/`. After login it
talks **directly** to Cognito and S3 from the browser — no server. Backend (Cognito,
Identity Pool, IAM, S3) is provisioned by Terraform in [`../backend`](../backend).

## How it fits together

- `src/` — source (Vite). `cognito.js` does SRP login, `storage.js` does the
  Identity-Pool → temporary-credentials → S3 dance, `main.js` wires up the UI.
- `config.js` — **generated by `terraform apply`** with the public pool/client/bucket
  ids. Committed and served (no secrets). The page won't work until this exists.
- `index.html` + `assets/` — **built output**, committed so GitHub Pages serves it.

The AWS SDK is **self-hosted** (bundled by Vite), because the site CSP is
`script-src 'self'` — CDN imports would be blocked.

## Build

```bash
cd portal
npm install
npm run build      # emits ./index.html and ./assets/ (committed for GitHub Pages)
```

Commit the regenerated `index.html` and `assets/` after a build.

## Local dev

```bash
npm run dev        # http://localhost:5173/portal/
```

Login is inert until `config.js` exists. For local testing after a `terraform apply`,
copy it next to the dev entry:

```bash
cp config.js src/config.js   # only needed for `npm run dev`
```

`http://localhost:5173` is already in the bucket CORS allow-list (see
`backend/variables.tf`).

## First use

1. `cd ../backend && terraform apply` (writes `config.js`).
2. Create yourself a user — see [`../backend/README.md`](../backend/README.md).
3. `npm run build`, deploy, open `/portal/`, sign in, set a permanent password, upload.

Files land at `s3://<bucket>/private/<your-identity-id>/…` and are only ever visible to
you (enforced by IAM, not the page).
