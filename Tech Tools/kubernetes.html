Core Concepts

The 60-Second Kubernetes Mental Model

Kubernetes is a distributed control system that keeps your application fleet close to the state you declared. You describe **intent** (YAML), and the cluster handles the **execution**.

1You apply a manifest (Intent).
2API Server validates & etcd stores it.
3Scheduler picks a Node for the Pod.
4Kubelet starts the containers.
5Controllers monitor & heal (Loop).
6Services provide stable access.

Control Plane (The Brain)

API Server

The entry point for kubectl and controllers.

etcd

Source of truth database for cluster state.

Scheduler

Decides which node gets which Pod.

Worker Nodes (The Muscle)

Kubelet

The node agent that ensures containers run.

Kube-Proxy

Handles Service networking and routing.

Container Runtime

Actually pulls images and runs containers.

Command atlas

The kubectl and kind cheat sheet you will actually use

Search by keyword or filter by category. The point is not just to list commands, but to explain what each one is for so you can reach for the right tool quickly under pressure.

33 rows
Command Impact Zone What it does When to use it
kind create cluster --name k8s-lab Local Creates a disposable local cluster with the name k8s-lab. Best starting point for laptop practice if you want a clean reset path.
kubectl cluster-info Control Plane Shows the control plane endpoints and confirms kubectl can reach the cluster. Run it immediately after cluster creation or when you think your context is wrong.
kubectl get nodes -o wide Control Plane Lists worker nodes and adds extra detail like internal IP and runtime information. Use it to confirm the cluster shape and whether nodes are Ready.
kubectl config get-contexts Local Shows every cluster context kubectl knows about. Useful when you are juggling local, staging, and production clusters.
kubectl config use-context <name> Local Switches kubectl to a different cluster context. Use this before every serious change so you do not touch the wrong cluster.
kubectl get ns Control Plane Lists namespaces. Good for orientation and for checking whether your target namespace exists yet.
kubectl create namespace lab Control Plane Creates a separate workspace called lab. Use namespaces to keep experiments from mixing with other workloads.
kubectl get pods -A Control Plane Shows every Pod across every namespace. Great when you do not yet know where the problem lives.
kubectl describe pod <pod> -n <ns> Control Plane Shows detailed Pod information, conditions, and recent events. First stop for Pending, restart loops, probe failures, and scheduling issues.
kubectl logs deploy/<name> -n <ns> --tail=100 Data Plane Shows recent logs from one Pod belonging to a Deployment. Use it when an app is starting but behaving strangely.
kubectl logs pod/<pod> -c <container> -f -n <ns> Data Plane Streams logs from a specific container inside a Pod. Use this for multi-container Pods or when you need live output during a test.
kubectl exec -it deploy/<name> -n <ns> -- sh Data Plane Opens an interactive shell in one container from the Deployment. Helpful for checking environment variables, DNS, mounted files, or app-local state.
kubectl apply -f app.yaml Control Plane Creates or updates resources from a manifest file. This is the declarative workhorse for repeatable changes.
kubectl diff -f app.yaml Local Shows what would change if you applied the manifest. Use it before apply when you want a quick safety check.
kubectl delete -f app.yaml Control Plane Deletes resources created from a manifest file. Best cleanup option when you used a file to create the workload.
kubectl scale deploy/<name> --replicas=5 -n <ns> Control Plane Changes the desired replica count of a Deployment. Use it to test horizontal scaling and self-healing behavior.
kubectl rollout status deploy/<name> -n <ns> Control Plane Watches a rollout until it completes or fails. Use it after changing an image or applying a new Deployment spec.
kubectl rollout history deploy/<name> -n <ns> Control Plane Shows rollout revisions for a Deployment. Useful when you need to understand what changed before rolling back.
kubectl rollout undo deploy/<name> -n <ns> Control Plane Rolls a Deployment back to the previous revision. Use it when a new image or config change breaks the app.
kubectl set image deploy/<name> <container>=nginx:1.27 -n <ns> Control Plane Updates the image for one container in a Deployment. Fast way to trigger and observe a rollout during a lab or emergency fix.
kubectl get svc,endpoints,ingress -n <ns> Control Plane Lists network entry points and the backend endpoints behind Services. Use it when traffic does not reach your Pods.
kubectl port-forward svc/<name> 8080:80 -n <ns> Local Maps a Service port to your local machine. Best fast path for testing a cluster service from your browser without exposing it publicly.
kubectl get pvc -n <ns> Control Plane Shows PersistentVolumeClaims and whether storage was bound successfully. Run this whenever a stateful workload is stuck waiting for storage.
kubectl create configmap app-config --from-literal=MODE=demo -n <ns> Control Plane Creates a ConfigMap from inline key-value data. Great for labs and for separating config from images.
kubectl create secret generic app-secret --from-literal=API_KEY=demo -n <ns> Control Plane Creates a Secret from inline key-value data. Use it for sensitive config that should not live in plain manifests.
kubectl get events -A --sort-by=.metadata.creationTimestamp Control Plane Shows cluster events in chronological order. Excellent for spotting image pulls, scheduling failures, or probe flaps across namespaces.
kubectl top pod -n <ns> Control Plane Shows CPU and memory usage for Pods if metrics-server is installed. Use it to confirm resource pressure before changing requests or limits.
kubectl explain deployment.spec.template.spec.containers Local Prints built-in documentation for a field path in the Kubernetes API. One of the best ways to learn the schema without leaving your terminal.
kubectl cordon <node> Worker Marks a node unschedulable so no new Pods land there. Use it before maintenance or while isolating a suspect node.
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data Worker Evicts movable Pods from a node to prepare it for maintenance. Use after cordoning when you need the workloads moved elsewhere.
kubectl uncordon <node> Worker Makes the node schedulable again. Use it after maintenance is complete.
kubectl delete namespace lab Control Plane Deletes the namespace and everything inside it. The safest large cleanup step for labs that were isolated inside one namespace.
kind delete cluster --name k8s-lab Local Removes the local kind cluster completely. Use this when you want a full reset and no leftover local cluster state.

Tip: pair kubectl describe, kubectl logs, and kubectl get events before you start guessing. Kubernetes usually tells you what it is unhappy about.

Lab 6

Persistence (PVCs)

Make data survive Pod restarts.

15 min
Apply
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: task-pv-claim
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 1Gi

Notice

A PVC allows you to request storage without knowing the underlying disk details.

Lab 8

Node Affinity

Control exactly where your Pods land.

15 min
Step 1: Label Node
kubectl label nodes  disktype=ssd
# In Deployment YAML:
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: disktype
          operator: In
          values: [ssd]

Goal

Ensure high-IOPS apps only run on nodes with SSDs.

Troubleshooting

A simple debugging flow beats panic every time

The mistake most beginners make is jumping straight into random fixes. Start with object state, then events, then logs, then networking, then deeper runtime inspection. The order matters because it cuts guesswork.

60-second debug sequence

1
Check the objects

kubectl get pods,svc,deploy -n app tells you whether the problem is existence, readiness, or routing.

2
Describe before you guess

kubectl describe pod <pod> -n app surfaces scheduling, image pull, probe, and mount errors.

3
Read logs

kubectl logs tells you whether the app crashed, misconfigured itself, or simply never started.

4
Check the network path

Compare Services, endpoints, DNS, and port-forward results before assuming the app is broken.

5
Inspect from inside

kubectl exec helps you verify files, env vars, service reachability, and DNS resolution from the workload itself.

6
Read events for timing

kubectl get events -A --sort-by=.metadata.creationTimestamp often reveals the exact moment things started failing.

Common signals and what they usually mean

Signal Likely meaning First move
Pending The Pod has not been scheduled or cannot mount required resources. kubectl describe pod
CrashLoopBackOff The container starts, fails, and keeps getting restarted. kubectl logs and probe review
ImagePullBackOff The runtime cannot pull the image, often due to a bad tag or auth issue. Check image name and registry credentials
Ready 0/1 The container is running but not healthy enough to receive traffic. Inspect readiness probes and app startup
Service has no endpoints The selector does not match healthy Pods. Compare Service selector with Pod labels
PVC Pending Storage has not been provisioned or bound yet. Check StorageClass and provisioner state
OOMKilled The container exceeded its memory limit. Review memory usage and resource limits

Cleanup

Always leave your local environment clean and boring

Good cleanup is part of learning. If you do not reset your namespaces, contexts, and local clusters, the next lab becomes harder to trust. Reverse the creation order whenever possible and delete at the highest safe boundary.

Fast cleanup checklist

  • Stop any active kubectl port-forward session with Ctrl+C.
  • Delete the namespace if all lab resources live there: kubectl delete namespace lab.
  • Delete file-based resources with kubectl delete -f file.yaml when you used manifests.
  • Reset the active namespace if you changed it: kubectl config set-context --current --namespace=default.
  • Delete the whole local cluster when you want a guaranteed fresh start: kind delete cluster --name k8s-lab.

What not to forget

  • Secrets and ConfigMaps survive until you delete them. They do not disappear just because a Pod did.
  • Stateful workloads may leave PVCs behind even after the app is gone, depending on reclaim policy and storage backend.
  • Namespaces delete asynchronously. Wait until they are actually gone before assuming the cluster is clean.
  • Context mistakes are real. Run kubectl config get-contexts before a destructive command if you work with multiple clusters.
Next Steps

Beyond the basics: Your production path

If you can manage the control plane, deploy applications, and handle rollbacks, you have the foundation. The next stage involves scaling these patterns with automation and GitOps.

Next practice targets

  • Add readiness and liveness probes to a real sample app.
  • Mount config through a ConfigMap and sensitive values through a Secret.
  • Run a StatefulSet with a PersistentVolumeClaim and verify data survives Pod replacement.
  • Introduce Helm after raw manifests feel readable.
  • Practice node maintenance with cordon, drain, and uncordon on a non-critical cluster.
